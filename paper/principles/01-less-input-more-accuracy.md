# Principle 1: Less Input, More Accuracy

An LLM's decision accuracy tends to be inversely related to the amount of context it must process - especially irrelevant context. An efficient agent and an accurate agent are often the same thing.
This principle is deceptively simple, widely understood in practice, and almost universally underappreciated in architecture.
Most practitioners who have worked with LLMs have encountered the basic phenomenon. As the context window fills - with conversation history, retrieved documents, system prompts, intermediate results - the model's ability to identify the relevant signal degrades. This is not a controversial claim. It is observable in even the most trivial cases: give an LLM an array of numbers and ask it to identify the highest value. The probability of a correct answer decreases as the array grows. The drop is negligible for small arrays, but it is real, it is measurable, and it scales.
The underlying mechanism is well-documented in the research literature under a growing family of formally defined terms: "context dilution" (progressive degradation of task-relevant signal as irrelevant tokens accumulate), "context pollution" (now used explicitly in Anthropic's engineering guidance for AI agents), and "context rot" (performance degradation that increases with input length regardless of content relevance). The foundational work is Liu et al.'s "Lost in the Middle" (2023, Stanford/UC Berkeley/Samaya AI), and subsequent studies have formalized the phenomenon rigorously - a NeurIPS 2025 workshop paper demonstrated a 47% drop in security feature implementation when moving from focused to heavily diluted contexts. I'll refer to this cluster of effects collectively as context dilution. An LLM processes context through attention - a mechanism that, in simplified terms, assigns weight to each piece of input relative to preceding context. As the volume of input grows, the model's ability to attend to the most relevant information degrades - through a combination of factors including the mathematical properties of softmax normalization (which forces attention weights to sum to 1, diluting focus as token count grows), positional encoding limitations (which degrade the model's ability to distinguish token positions at length), and phenomena like attention sinks (where initial tokens absorb disproportionate weight regardless of content). Irrelevant tokens do not simply sit inert in the context window. They contribute to this degradation, making it harder for the model to isolate and prioritize the signal. The result is that the signal does not merely fail to increase with more input. It degrades, because the model's finite capacity for focused attention is diluted by noise.
This has a counterintuitive architectural implication: efficiency and accuracy are not tradeoffs. They are correlated. In most engineering disciplines, optimization for speed or cost involves some sacrifice in quality. You cut corners to go faster. You reduce fidelity to reduce expense. In agentic AI, the relationship is inverted. A tighter context window - fewer tokens, less irrelevant data, more precisely curated input - is simultaneously cheaper to process and more likely to produce a correct result. The optimization that reduces cost is the same optimization that increases quality. In the electricity analogy: a well-designed appliance doesn't just use less power - it does better work because it uses less power. Wasted energy is wasted signal. The MemAct framework (arXiv, revised January 2026) demonstrates this at scale: by actively curating context during long-horizon agentic tasks, it achieves accuracy matching models 16× larger while reducing average context length by 51%.
This is not a minor observation. It has direct consequences for how agentic systems should be designed:
Retrieval should be precise, not comprehensive. The instinct in many enterprise implementations is to retrieve as much potentially relevant data as possible and let the LLM sort through it. This is architecturally equivalent to dumping an entire filing cabinet on someone's desk and asking them to answer a specific question. A well-designed retrieval layer - and this is one of the primary functions of the Agentic Interface Layer described in Principle 3 - should return the minimum context necessary to answer the question, not the maximum context that might be tangentially related.
Conversation history should be managed, not accumulated. Many agentic implementations pass the entire conversation history to the model on every turn. This means that by the twentieth interaction, the model is processing nineteen turns of potentially irrelevant prior context alongside the current request. Architectural patterns for history summarization, selective retention, and context windowing are not optimizations to be added later. They are accuracy requirements that should be present from the start.
System prompts should be lean. There is a tendency to front-load system prompts with every possible instruction, guardrail, persona specification, and edge case handler. Each addition feels individually justified. Collectively, they create a dense preamble that the model must attend to on every single request, regardless of relevance. The system prompt is not free context. It is context that competes with the actual task for attention.
Multi-step orchestration should minimize context carried between steps. When an orchestrating agent calls a sub-agent or skill, the temptation is to pass along "everything it might need" - the full history, the full retrieval, the full state. Each step should receive only what it requires for its specific function. The orchestrator's job is not just to decide what to call, but to curate what context to pass.
A Clarifying Example
A scenario I encountered illustrates both the importance and the nuance of this principle.
The task was to translate a large mapping codebase - a mix of XSLT and a native scripting language - into pure SQL. The initial approach followed what appeared to be sound context minimization: go field by field, have an LLM identify how each field is mapped, generate the SQL for that field, then collate all the individually generated SQL statements and have an LLM aggregate them into a single query.
The second approach ignored context minimization entirely. The full mapping document was fed to the LLM, and it was asked to generate the holistic SQL translation in a single pass.
The second approach produced results that were better than the field-by-field method.
This appears to contradict the principle. It does not. It refines it.
A mapping document is relationally dense. Fields do not exist independently. They reference each other, share transformation logic, carry implicit dependencies, and follow structural patterns that are only visible when the full document is present. Isolating a single field does not remove irrelevant context - it removes relevant context. The relationships between fields are not noise.
They are signal that the model needs in order to understand the mapping logic as a coherent system rather than a collection of independent translations.
The field-by-field approach also introduced an additional problem: aggregation. The individually generated SQL fragments were each produced without awareness of the others. Merging them into a coherent whole required the LLM to reconstruct relationships that it was never given the context to understand - a task arguably harder than the original translation, because the relational signal had been deliberately stripped out.
The holistic approach worked better because the full document was the minimum relevant context. The signal-to-noise ratio was higher with the complete mapping than with isolated fragments, because the structure itself carried meaning.
This clarifies what the principle is actually about. "Less input" does not mean "smaller documents" or "fewer tokens in absolute terms." It means less irrelevant input - less noise competing with signal for the model's attention. A 10,000-token document where every token is structurally relevant to the task is a tighter context than a 500-token excerpt that has been stripped of the relational information needed to understand it. The unit of optimization is not token count. It is signal density.
Tradeoffs and Limitations
This principle, applied carelessly, can be used to justify withholding context that the model genuinely needs. The goal is not minimal context in absolute terms - it is minimal irrelevant context. The distinction matters. There are tasks that legitimately require large context windows: legal document analysis, multi-system reconciliation, long-form synthesis. The principle does not argue against large context when the task demands it. It argues against large context when the architecture is simply too lazy or too uncurated to filter what the task actually needs.
The practical challenge is that determining what is "relevant" requires intelligence - often the same intelligence you are trying to feed the context to. This creates a bootstrapping problem: you need to understand the question to know what context is relevant, but you need the context to understand the question. Architectural solutions to this - staged retrieval, query decomposition, progressive context refinement - exist but add complexity. The principle does not eliminate this tension. It simply insists that the tension be addressed architecturally rather than ignored by dumping everything into the window and hoping for the best.
There is also a practical floor. Some minimum context is always required, and over-optimization can produce an agent that is efficient but insufficiently informed. The right mental model is not
"as little as possible" but "as little as possible while still sufficient for the task." The Agentic Interface Layer, when well-designed, is the architectural mechanism that makes this balance achievable - it curates context so that the agent does not have to choose between having enough information and having too much.
