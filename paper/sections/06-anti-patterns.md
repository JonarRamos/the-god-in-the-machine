# Anti-Patterns

A framework becomes teachable when people can identify not only what good looks like, but what bad looks like. SOA became widely adoptable partly because practitioners could point at a system and say "that's not SOA, that's a distributed monolith." The vocabulary of failure was as useful as the vocabulary of success.
The following anti-patterns are the most common failure modes I have observed in enterprise agentic implementations. Each one is a violation of one or more Theotropic Architecture principles, and each one produces systems that work in demo and degrade in production.
The Fat Agent
What it looks like: Everything is in the prompt. The system prompt is thousands of tokens long, containing every instruction, every guardrail, every edge case, every persona specification, every piece of reference data the agent might conceivably need. The conversation history is passed in full on every turn. Retrieved documents are dumped into the context wholesale, unfiltered and uncurated. The agent is a single monolithic entity that knows everything, holds everything, and processes everything in a single context window.
Why it happens: It is the path of least resistance. Adding context to the prompt is easy. Curating context requires architecture. When a new requirement emerges - "the agent should also know about X" - the simplest response is to add X to the prompt. Over time, the prompt accumulates until the agent is processing tens of thousands of tokens on every invocation, most of it irrelevant to the current request.
Why it fails: Principle 1. Irrelevant context degrades accuracy. The Fat Agent is not just inefficient - it tends to be less accurate than a thinner version of itself. It hallucinates more because the signal is diluted. It costs more because it processes more tokens. It is slower because the context window is larger. And it is fragile because any change to any piece of the accumulated context can affect the agent's behavior in unpredictable ways.
The Theotropic alternative: A thin agent with a curated Agentic Interface Layer, composable skills, and a memory layer that provides only what is needed for the current request. The context is assembled dynamically based on the task, not accumulated statically in the prompt.
Blind Orchestration
What it looks like: The agent operates, but nobody can see what it is doing or why. Decisions are not logged. Skill invocations are not recorded. When the agent produces an unexpected result, there is no way to reconstruct the reasoning path. The system is a black box - inputs go in, outputs come out, and the space between is invisible.
Why it happens: Observability is infrastructure work that does not produce visible features. It is easy to defer - "we'll add logging later" - especially under pressure to deliver a working agent quickly. The agent works in demo without logging. It works in early production without logging. The absence of observability is not felt until something goes wrong and there is no way to diagnose it.
Why it fails: Principle 9. Without observability, there are no feedback loops. Without feedback loops, there is no systematic improvement. The system is as good on day one thousand as it was on day one. Failures are opaque. Debugging is guesswork. Compliance teams cannot audit the system's decisions. And the compounding improvement that makes the framework's maturity model possible - each cycle of logging, analysis, and correction making the system better - never begins.
The Theotropic alternative: Comprehensive logging from the first implementation. Every decision, every skill call, every retrieval, every self-healing event is recorded. The investment is front-loaded, but the return is cumulative - every logged execution is a data point for future improvement.
Pipeline Thinking
What it looks like: The agentic workflow is designed as a linear sequence. Step 1 retrieves data. Step 2 processes it. Step 3 generates a result. Step 4 delivers it. The sequence is fixed. If Step 2 reveals that Step 1 retrieved the wrong data, the pipeline either fails or restarts from the beginning. There is no mechanism for the agent to revisit earlier steps, call skills out of order, or adapt the execution path based on intermediate results.
Why it happens: Pipelines are familiar. Every workflow automation tool, every ETL process, every CI/CD system is built on the pipeline model. When practitioners come to agentic AI, they bring the pipeline mental model with them. It feels natural to design an agent as a sequence of steps, because that is how most automation has been designed for decades.
Why it fails: Principle 6. An agent pursuing an outcome (Principle 4) does not think in sequences. It thinks in goals. It needs to retrieve, evaluate, adjust, re-retrieve, try a different approach, and iterate until the outcome is achieved. A pipeline constrains the agent to a predetermined path, which limits the adaptive intelligence that makes agentic systems valuable. The result is an agent that has the cost profile of an LLM-based system but the flexibility of a traditional workflow engine.
The Theotropic alternative: Composable, recursively callable skills with a thin orchestrating agent that determines the execution path dynamically. The agent is free to call any skill in any order, revisit previous steps, and iterate until the outcome is satisfied - within the safeguards (depth limits, cost ceilings, cycle detection) that make recursive orchestration production-safe.
Keyword Roulette
What it looks like: Routing is based on keyword matching. If the user's request contains "invoice," route to the invoicing skill. If it contains "employee," route to the HR data. The routing logic is a decision tree of string matches, regular expressions, or simple classification rules tied to specific words.
Why it happens: Keyword matching is simple to implement and easy to understand. It works well enough in controlled demos where the test inputs are crafted to match the expected keywords. It also maps to how traditional chatbots were built - intent classification based on keyword triggers - so practitioners with chatbot experience tend to default to it.
Why it fails: Principle 7. Natural language is not a controlled vocabulary. Users say "bill" when they mean "invoice." They say "headcount" when they mean "employees." They phrase the same request in ten different ways. Keyword routing is brittle - it breaks on synonyms, paraphrases, and novel phrasing. It also fails silently - when no keyword matches, the system either guesses or fails, with no graceful mechanism for disambiguation. As the skill registry grows and the number of possible routes increases, the keyword decision tree becomes a maintenance nightmare of overlapping and conflicting rules.
The Theotropic alternative: Intent-based routing. Convert incoming requests to semantic representations and match against semantically described skills. Cache resolved intents for performance. The routing is robust to paraphrasing, extensible to new skills without rewriting decision trees, and improvable through the feedback loops of Principle 9.
The Disposable Agent
What it looks like: Each new AI use case gets its own agent, its own data pipeline, its own skills, its own error handling, and its own infrastructure. Nothing is shared. Nothing is reusable. The organization builds ten agents and has ten completely independent systems with no architectural relationship to each other.
Why it happens: It is faster to build from scratch than to build for reuse. The first agent does not justify the investment in shared infrastructure. The second agent is urgent and cannot wait for the shared layer to be ready. By the third and fourth agents, the pattern is established - each one is a standalone project. The cost of unification grows with each new agent, making it increasingly expensive to course-correct.
Why it fails: Principles 3, 6, and 10. Nothing compounds. Each agent is a silo. The eleventh agent starts from zero, with no benefit from the ten that preceded it. The organization is building tactical AI - solving individual problems - without any strategic architecture connecting them. The cost dynamics of this pattern - and why Theotropic Architecture's compounding alternative is more favorable over time - are examined in the maturity model.
The Theotropic alternative: Build each agent against shared infrastructure from the start. The Agentic Interface Layer is shared. The skill registry is shared. The feedback loops are shared. Each agent contributes to the platform rather than standing alone.
The Unforgiving System
What it looks like: Errors are terminal. When a skill fails, the entire operation fails. The agent receives a generic error message - or no message at all - and has no mechanism to adapt, retry with corrected inputs, or route around the failure. The user sees "Something went wrong. Please try again." Why it happens: Error handling is treated as an edge case rather than a core design concern. Skills are built for the happy path. Failure responses are generic because specific failure analysis requires additional development effort. The orchestrating agent is not designed to interpret failures because the skills do not provide interpretable failure information. OpenClaw's GitHub issue #6823 - a feature request for execution guardrails - illustrates the consequence: it cites an incident where an agent deleted OAuth credentials while "trying to fix an auth issue," motivating the request for safety mechanisms that, with structured error responses, would have made the failure a correctable retry rather than a destructive action.
Why it fails: Principle 8. In enterprise environments, failures are not edge cases - they are routine. Data is inconsistent. Systems are intermittently unavailable. Formats don't match. Identifiers conflict. A system that cannot handle routine failures gracefully will have an unacceptable failure rate in production, regardless of how well it performs on the happy path. In my experience, the difference between a moderate success rate and a near-complete one is often nothing more than structured failure handling.
The Theotropic alternative: Skills return structured failure responses - what went wrong, why, and how to fix it. The orchestrating agent interprets these responses and adapts within the same execution. The system does not fail at the first obstacle. It diagnoses, corrects, and retries. The user sees a successful result. The logs show the recovery path.
The Oracle
What it looks like: The agent is given direct access to production systems - production databases, live APIs, real-time transaction systems - with no intermediary layer. It queries production directly. In later maturity stages, it writes to production directly. There is no abstraction, no curation, and no blast radius isolation. OpenClaw is the canonical example at consumer scale: a single process with direct access to email, calendars, file systems, messaging platforms, and shell commands, with no intermediary layer between the agent's decisions and the real systems it acts on. The results were impressive - and the blast radius was the user's entire digital life.
Why it happens: It is the fastest path to a working demo. Connecting the agent directly to production data avoids the effort of building an intermediary layer. The results are impressive - the agent is working with real, current data. OpenClaw accumulated 247,000 GitHub stars precisely because it delivered on this promise. The risk is invisible until something goes wrong - and with OpenClaw, it went wrong publicly and repeatedly: deleted inboxes, erased credentials, hundreds of unsanctioned messages, and a security firm finding over 30,000 exposed instances in a 12-day analysis period.
Why it fails: Principle 3. Without the Agentic Interface Layer, every agent failure has the potential to affect production systems. A malformed query can degrade database performance. A write operation with incorrect data can corrupt production records. A permission misconfiguration can expose sensitive data. The blast radius is the production environment - which is to say, there is no blast radius isolation at all.
The Theotropic alternative: The Agentic Interface Layer. The agent operates against a curated abstraction. Production systems are insulated. Failures are contained. The data the agent works with is specifically prepared for agentic consumption, which also improves accuracy (Principle 1) and reduces the agent's complexity (Principle 2). For write operations, the layered defense model described in Principle 3 applies: changes are staged in the AIL before reaching production, human gates provide approval checkpoints for critical writes, and deterministic validation in the write skills themselves provides a final mechanical safeguard. No single layer is sufficient. Together, they ensure that the agent's reach never exceeds its governance.
These anti-patterns are not theoretical. They are the default outcome when agentic systems are built without architectural principles. Most organizations currently building AI agents are, in my observation, exhibiting at least two or three of them simultaneously. Recognizing them is the first step toward the architectural discipline that Theotropic Architecture proposes.
